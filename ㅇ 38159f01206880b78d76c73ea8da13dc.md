- 패키지 구조
    
    ```
    domain/{domain_name}
    ├── presentation
    │   └── api
    │       ├── request
    │       └── response
    ├── application
    │   ├── command
    │   ├── policy
    │   ├── port
    │   ├── service
    │   └── usecase
    ├── domain
    │   ├── model
    │   ├── policy
    │   └── repository
    └── infrastructure
        ├── persistence
        └── {외부 기능 또는 참조 목적별 패키지}
    ```
    
    > 현재 강의 도메인의 실제 패키지명은 course가 아니라 cource입니다. 기존 코드의 import 경로와 일치시키기 위해 현재 문서에서도 cource를 사용합니다.

- 성공 응답 포맷
    
    성공 응답은 `global.common.ApiResponse<T>`를 사용합니다.
    
    ```java
    // 조회 / 수정 (200)
    return ApiResponse.success(
        "강의가 조회되었습니다.",
        response
    );
    
    // 생성 (201)
    return ApiResponse.created(
        "강의가 등록되었습니다.",
        response
    );
    
    // 반환 데이터가 없는 성공 응답 (200)
    return ApiResponse.successNoContent(
        "영상 완료 처리되었습니다."
    );
    ```
    
    **조회 / 수정 성공 응답 포맷**
    
    ```json
    {
      "httpStatus": 200,
      "message": "강의가 조회되었습니다.",
      "data": {
        "courseId": 1,
        "title": "Spring Boot 강의"
      }
    }
    ```
    
    **생성 성공 응답 포맷**
    
    ```json
    {
      "httpStatus": 201,
      "message": "강의가 등록되었습니다.",
      "data": {
        "courseId": 1
      }
    }
    ```
    
    **반환 데이터가 없는 성공 응답 포맷**
    
    ```json
    {
      "httpStatus": 200,
      "message": "영상 완료 처리되었습니다.",
      "data": null
    }
    ```
    
- 에러 응답 처리
    
    **1. 공통 처리 방식**
    
    비즈니스 로직 처리 중 발생하는 예외는 BusinessException에 ErrorCode를 전달하여 발생시킵니다.
    
    ```java
    throw new BusinessException(ErrorCode.USER_NOT_FOUND);
    ```
    
    GlobalExceptionHandler가 예외를 처리하고 ErrorResponse를 반환합니다.
    
    ```
    BusinessException
      -> GlobalExceptionHandler
        -> ErrorResponse
    ```
    
    **2. ErrorCode 구조**
    
    전역 에러 코드는 `global.exception.ErrorCode` enum에 정의합니다.
    
    ```java
    USER_NOT_FOUND(
        HttpStatus.NOT_FOUND,
        "U001",
        "가입되지 않은 이메일입니다."
    )
    ```
    
    | 필드 | 의미 및 설명 |
    | --- | --- |
    | status | HTTP Status |
    | code | 클라이언트에 전달할 에러 코드 |
    | message | 클라이언트에 전달할 에러 메시지 |
    
    **3. 현재 에러코드 접두어**
    
    | 도메인 | 접두어 | 예시 |
    | --- | --- | --- |
    | 공통 | C | C001, C002 |
    | 회원 | U | U001, U002 |
    | 인증 및 보안 | AUTH_ | AUTH_002, AUTH_009 |
    | 강의 | CR | CR001, CR004 |
    | 리뷰 | R | R001, R004 |
    | 수강 | EN | EN001 |
    | 학습 활동 | L | L001, L004 |
    | 게시글 | P | P001, P006 |
    | 파일 | F | F001, F004 |
    | 댓글 | C | C001, C009 |
    | 공지사항 | N | N001, N004 |
    
    **4. 에러코드 추가 방법**
    
    1. `global.exception.ErrorCode` enum에 상수를 추가합니다.
    2. HTTP Status, 에러 코드, 에러 메시지를 함께 작성합니다.
    3. 비즈니스 로직에서 `throw new BusinessException(ErrorCode.X)` 형식으로 사용합니다.
    4. 동일한 코드 문자열이 이미 존재하는지 확인합니다.
    
    **5. 에러 응답 포맷**
    
    ```json
    {
      "errorCode": "U001",
      "message": "가입되지 않은 이메일입니다.",
      "timestamp": "2026-06-01T10:00:00",
      "path": "/api/auth/login",
      "traceId": "trace-id-value",
      "details": null
    }
    ```
    
    **6. GlobalExceptionHandler 처리 대상**
    
    | 예외 | HTTP Status | 용도 |
    | --- | --- | --- |
    | MethodArgumentNotValidException | 400 | @Valid 요청값 검증 실패 |
    | IllegalArgumentException | 400 | 도메인 규칙 위반 |
    | BusinessException | ErrorCode에 정의된 값 | 비즈니스 로직 예외 |
    | Exception | 500 | 처리되지 않은 시스템 예외 |

- Swagger 작성 가이드
    
    의존성: `org.springdoc:springdoc-openapi-starter-webmvc-ui:2.8.16`
    
    Swagger UI: `http://localhost:8080/swagger-ui/index.html`
    
    **새 엔드포인트 작성 예시**
    
    ```java
    @Tag(name = "Learning Activity", description = "영상 재생, 학습 진도, 학습 활동 기록 API")
    @RestController
    @RequestMapping("/api/learning/videos")
    @RequiredArgsConstructor
    public class VideoProgressController {
    
        @Operation(
            summary = "영상 시청 시간 누적 저장",
            description = "영상 시청 중 추가로 시청한 시간을 초 단위로 누적 저장합니다."
        )
        @PatchMapping("/{videoId}/progress/watch-time")
    
            return ApiResponse.successNoContent("영상 시청 시간이 누적 저장되었습니다.");
        }
    }
    ```
    
    **규칙**
    
    - Controller에 `@Tag`를 작성합니다.
    - Endpoint에 `@Operation`을 작성합니다.
    - URL 식별자에는 필요한 경우 `@Parameter`와 검증 애노테이션을 작성합니다.
    - Request DTO 필드에는 필요한 경우 `@Schema`와 Jakarta Validation 애노테이션을 작성합니다.
    - 공통 예외 응답은 GlobalExceptionHandler에서 처리합니다.

    | Controller 패키지 | 일부 도메인은 presentation.api, 일부는 presentation을 사용합니다. |
    | Request / Response 패키지 | 일부 도메인은 presentation.api.request, 일부는 presentation.request를 사용합니다. |
    | JPA Repository 이름 | XxxJpaRepository와 SpringDataXxxRepository 형식이 함께 존재합니다. |
    | Repository Adapter 이름 | XxxRepositoryAdapter와 XxxPersistenceAdapter 형식이 함께 존재합니다. |
    | 성공 응답 | 별도 성공 코드 파일 없이 ApiResponse<T>를 사용합니다. |
    | 에러 코드 | 전역 ErrorCode enum을 사용합니다. 추가 시 기존 코드 문자열과 중복되지 않는지 확인합니다. |

- prod / dev 환경 차이
    
    [프로젝트 내 prod, dev 차이 상세](https://app.notion.com/p/371649136c11808b8cd5fa8096596c70?pvs=21)
    
    **1. 콘솔 출력 여부**
    
    - **dev** : 모든 로거에 `CONSOLE` appender가 붙어 있어 실시간으로 로그를 눈으로 확인 가능
    - **prod** : `CONSOLE` 출력 제거. 오직 파일(`FILE`, `ERROR_FILE`)로만 기록. 콘솔 출력이 서버 자원을 소모하므로 제거
    
    **2. 하이버네이트(JPA) 쿼리 노출 여부**
    
    - **dev** : `org.hibernate.SQL`이 `DEBUG`, `BasicBinder`가 `TRACE`로 설정. SQL 쿼리 및 파라미터 바인딩 값까지 전부 출력
    - **prod** : `org.hibernate` 전체가 `WARN`. 쿼리 로깅은 디스크 풀 장애와 성능 저하의 주범이므로 에러 시에만 기록
    
    **3. AOP 성능 로깅 (LoggingAspect)**
    
    - **dev** : `LoggingAspect`가 `INFO` 레벨. 메서드 진입/종료 로그를 통해 전체 흐름 파악 가능
    - **prod** : `LoggingAspect`가 `WARN` 레벨. 정상 로직은 로그 미발생. 메서드 실행 시간이 1초(1000ms) 초과 시에만 WARN 로그 발동 → 병목 구간만 탐지
    
    **4. Root 로거(스프링 프레임워크) 침묵**
    
    - **dev** : Root가 `INFO`, `org.springframework.web`이 `DEBUG`. 스프링 부팅 및 컴포넌트 스캔 과정 전부 출력
    - **prod** : Root가 `WARN`. 프레임워크 및 외부 라이브러리의 자잘한 정보성 로그를 음소거하고 비즈니스 로그와 에러 로그만 기록
- 예외 상세 처리 구조
    
    **1. details Map 생성**
    
    `Map<String, Object> details = new HashMap<>();` : 에러 내역을 담을 빈 바구니(Map)를 생성. JSON 변환 시 `{"필드명": "에러 메시지"}` 형태로 출력
    
    **2. e.getBindingResult()**
    
    `BindingResult`는 데이터 바인딩 및 유효성 검사 결과를 담은 보고서 객체. 유효성 검사 통과 여부 및 에러 정보 포함
    
    **3. .getFieldErrors()**
    
    결과 보고서 안에서 특정 DTO 필드(예: `email`, `password`)에서 발생한 에러만 `List<FieldError>` 형태로 추출
    
    **4. .forEach(error → ...)**
    
    에러 리스트를 순회하며 하나씩 처리. 예: 이메일과 비밀번호 모두 오류 시 로직을 2회 실행
    
    **5. details.put(error.getField(), error.getDefaultMessage())**
    
    - `error.getField()` : 유효성 검사에 실패한 필드명 반환 (예: `"email"`, `"score"`)
    - `error.getDefaultMessage()` : 어노테이션에 설정한 에러 메시지 반환 (예: `@NotBlank(message = "이메일은 필수입니다")` → `"이메일은 필수입니다"`)
- @PreAuthorize
    
    팀에 뿌릴 짧은 규칙
    권한 어노테이션 컨벤션
    
    1. 모든 권한 제한 API에는 @PreAuthorize를 사용한다.
    2. ROLE_ 접두사는 어노테이션에 쓰지 않는다.
    O: @PreAuthorize("hasRole('ADMIN')")
    X: @PreAuthorize("hasRole('ROLE_ADMIN')")
    3. 학생 이상 접근 API는 hasRole('STUDENT')를 사용한다.
    4. 강사 이상 접근 API는 hasRole('INSTRUCTOR')를 사용한다.
    5. 관리자 전용 API는 hasRole('ADMIN')를 사용한다.
    6. 권한 계층은 아래와 같다.
    ADMIN > INSTRUCTOR > STUDENT
    7. URL 권한 설정과 @PreAuthorize가 충돌하지 않게 SecurityConfig에서는 공통 인증 여부만 관리하고, 세부 권한은 메서드에서 관리한다.

    ## **심는 위치 컨벤션**
    
    **레이어별 역할 분리:**
    
    - **Application(UseCase/Service)** — 비즈니스 결과 메트릭 (성공/실패 카운터)
    - **Infrastructure(Adapter)** — 외부 시스템 연동 메트릭 (캐시 hit/miss, Redis 지연)
    - **Presentation(Controller)** — API 응답시간은 Actuator가 자동 수집하므로 직접 심지 않아도 됨
    
    `// Application 레이어 예시 (심는 방식)@Service@RequiredArgsConstructorpublic class NoticeQueryService {
        private final MeterRegistry meterRegistry;
        public NoticeListResponse getNotices(...) {        // Cache Hit        meterRegistry.counter("notice.cache.hit.count").increment();                // Cache Miss        meterRegistry.counter("notice.cache.miss.count").increment();    }}`
